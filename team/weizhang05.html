PROJECT: FOP Manager

Overview

FOP Manager is a desktop application for NUS undergraduate students who are Project Directors of Freshmen Orientation Programs to manage both participants and committee members of the program. The user interacts with it using a CLI, and it has a GUI created with JavaFX. It is written in Java, and has about 10 kLoC. The development of this application is to teach Principles of Software Engineering.

Randomized Group Allocation

FOP Manager has a randomize command, which allows the Project Director to automatically assign all participants to a group.

Following is the activity diagram when the command is executed:

Current Implementation

The randomize is achieved by coding a sequence of steps to achieve the allocation of groups to participants: * Creating list holding specific type of participants (i.e. all OGLs will be stored in a list while freshmen will be stored in another) * Shuffles all of the previously created list * Assigns all of the participants to a group in order of freshmen, OGLs

Design Considerations

Aspect: Updating group attribute of participants
Alternatives Pros Cons

Programmatically call the edit command

Changing of interface with the model only requires changing the EditCommand and Parser

Tightly coupled with EditCommand and Parser

Interact with ModelManager (current choice)

Modification is obvious and follow guidelines of the architecture of the system and cleaner code

Coupled with ModelManager

Rationale: Choice is made based on design practice. It may seems very intuitive to programmatically call the edit command, but it might causes unknown complexity within the system if there is a change in the EditCommand or the parser.

Aspect: Shuffling of participants
Alternatives Pros Cons

Creating lists for OGLs and freshmen and assigning them list by list (current choice)

Ensures all groups will have at least an OGL and balanced distribution amongst the group

Requires more memory space

Rationale: There is only 1 consideration for this feature due to it’s complexity, no other alternative has yet been identified. Future developers are free to modify the algorithm but must ensure all groups has at least an OGL and freshmen are evenly distributed (with 1 or 2 difference in numbers for freshmen).
